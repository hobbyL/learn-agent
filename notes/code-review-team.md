# 代码审查团队：分工协作模式

> 基于项目 16 实践整理

---

## 1. 协作模式对比

多 Agent 系统的两种基础协作模式：

### 辩论投票模式（项目 15）

```
目标：达成共识
结构：平等辩论
信息流：交叉质疑
输出：投票结果

        问题
         ↓
    ┌────┼────┐
   观点A 观点B 观点C
    └────┼────┘
      交叉质疑
         ↓
       投票
```

**适用场景**：决策问题、方案选择、优先级排序

### 分工协作模式（项目 16）

```
目标：全面覆盖
结构：角色分工
信息流：独立执行
输出：聚合报告

        任务
         ↓
    ┌────┼────┐
   维度A 维度B 维度C
    └────┼────┘
      并行执行
         ↓
       汇总
```

**适用场景**：多维度分析、全面审查、并行处理

## 2. 两层协作架构

```
┌─────────────────────────────────────────┐
│  执行层：专业 Agent 并行                 │
│  每个 Agent 负责一个维度，独立执行       │
│  输出：结构化数据（json_schema）         │
└─────────────────────────────────────────┘
                 ↓
┌─────────────────────────────────────────┐
│  决策层：汇总 Agent                      │
│  去重、排序、优先级判断                  │
│  输出：自然语言报告                      │
└─────────────────────────────────────────┘
```

### 关键设计点

1. **执行层为什么独立？**
   - 避免信息干扰：每个 Agent 只看到原始任务，不受其他 Agent 影响
   - 维度清晰：安全审查不需要知道性能审查的结果
   - 可并行：独立意味着可以用 asyncio 真正并行执行

2. **决策层为什么必要？**
   - 去重：多个 Agent 可能发现同一问题
   - 排序：P0 致命问题 > P1 严重问题 > P2 优化建议
   - 综合判断：跨维度的问题关联（如安全漏洞可能导致性能问题）

3. **为什么执行层用结构化、决策层用自然语言？**
   - 执行层输出需要被程序解析（去重、排序）
   - 决策层输出面向人类（最终报告）

## 3. 结构化输出的工程价值

### 审查员输出（json_schema 强制）

```json
{
  "reviewer": "安全审查员",
  "findings": [
    {
      "file": "user_auth.py",
      "line": 12,
      "severity": "P0",
      "category": "sql_injection",
      "description": "...",
      "suggestion": "..."
    }
  ],
  "summary": "发现 5 个高危问题",
  "pass_review": false
}
```

**为什么必须结构化？**
- 主审需要精确解析：文件名、行号、严重级别
- 去重依赖精确匹配（文件名 + 行号）
- 排序依赖结构化字段（severity）

**如果用自然语言会怎样？**
- "在 user_auth.py 的第 12 行发现 SQL 注入"
- "user_auth.py:12 存在注入风险"
- 同一问题，不同表述，无法精确去重

### 主审输出（自然语言）

```
=== 代码审查最终报告 ===

【概览】
4 位审查员共发现 12 个问题，去重后剩余 9 个：
- P0 致命：3 个
...
```

**为什么可以自然语言？**
- 最终报告不需要再次解析
- 面向人类开发者，自然语言更易读
- 总结性质，不需要精确的程序化处理

## 4. 去重逻辑的实现

### 方案一：规则匹配（未采用）

```python
def deduplicate(findings):
    seen = set()
    unique = []
    for f in findings:
        key = (f.file, f.line)
        if key not in seen:
            seen.add(key)
            unique.append(f)
    return unique
```

**问题**：
- 同一问题可能跨多行（如一个函数的设计问题）
- 不同位置可能是同一类问题（如多处 SQL 注入）

### 方案二：语义去重（项目采用）

主审 Agent 用 LLM 理解问题描述，判断语义相似性：

```python
# 输入：4 个审查员的所有 findings
# 输出：主审自己判断哪些是重复的，输出去重后的报告
```

**优势**：
- 灵活：可以识别不同表述的同一问题
- 智能：可以跨维度去重（安全问题 + 架构问题可能是同一根因）

**劣势**：
- 不可控：去重逻辑在 LLM 黑盒里
- 成本：多一次 API 调用

## 5. 虚构代码库的设计原则

### 原则一：单一维度集中

```python
# 代码 A：主要安全问题
CODE_A = '''
API_KEY = "sk-..."  # 硬编码密钥
query = f"SELECT * FROM users WHERE username='{username}'"  # SQL 注入
'''

# 代码 B：主要性能问题
CODE_B = '''
for order in orders:
    for item in order['items']:  # O(n²)
        time.sleep(0.1)  # 阻塞 I/O
'''
```

**为什么不混合？**
- 方便验证：代码 A 的安全问题应该被安全审查员找到
- 避免维度重叠：如果每段代码都有安全+性能问题，无法测试"分工"

### 原则二：问题可验证

```python
preset_issues = {
    "security": [
        {"line": 12, "severity": "P0", "type": "sql_injection", "desc": "..."}
    ]
}
```

预设问题标注：
- 准确行号
- 严重级别
- 问题类型

这样可以量化召回率：`找到的问题数 / 预设问题数`

### 原则三：覆盖典型场景

```
安全：SQL 注入、硬编码密钥、权限缺失
性能：O(n²)、重复计算、阻塞 I/O
架构：职责不清、耦合过紧、错误处理缺失
规范：命名混乱、缺文档、代码重复
```

这些是真实代码审查中最常见的问题类型。

## 6. 召回率验证

```python
preset_count = {"security": 5, "performance": 5, ...}
found_count = {"security": 4, "performance": 3, ...}

recall = found / preset * 100  # 召回率
```

**意义**：
- 验收标准：召回率 ≥ 80% 说明审查员有效
- 调优依据：召回率低的维度需要优化 system prompt

**局限**：
- 只能验证"预设问题"，无法验证审查员发现的"额外问题"
- 预设问题的设计质量影响评估准确性

## 7. 适用场景

分工协作模式适合：

| 场景 | Agent 分工 |
|------|-----------|
| 代码审查 | 安全 / 性能 / 架构 / 规范 |
| 内容审核 | 违规 / 低俗 / 广告 / 版权 |
| 市场调研 | 竞品 / 用户 / 技术 / 政策 |
| 文档生成 | API / 架构 / 部署 / FAQ |
| Bug 分析 | 前端 / 后端 / 数据库 / 网络 |

**核心特征**：
- 任务可拆分为多个独立维度
- 每个维度需要专业知识
- 最终需要综合所有维度的结果

## 9. 实现建议

### 1. 执行层要真正独立

❌ **错误**：审查员 B 看到审查员 A 的结果
```python
# 审查员 B 的 prompt
f"审查员 A 发现了 {len(results_A)} 个问题，你负责性能维度..."
```

✅ **正确**：每个审查员只看到原始代码
```python
# 审查员 B 的 prompt
"你是性能审查专家，请审查以下代码..."
```

### 2. 决策层要有权威

主审不是"把 4 份报告拼在一起"，而是：
- 去重（判断哪些是同一问题）
- 排序（判断优先级）
- 综合（跨维度关联分析）

### 3. 结构化输出要 strict

```python
response_format={
    "type": "json_schema",
    "json_schema": {
        "strict": True,  # 必须
        "schema": schema
    }
}
```

不 strict = 可能不符合 schema = 主审解析失败。

## 10. 扩展方向

1. **真正的并行执行**
   ```python
   results = await asyncio.gather(
       review_code(code, reviewer_A),
       review_code(code, reviewer_B),
       review_code(code, reviewer_C),
       review_code(code, reviewer_D)
   )
   ```

2. **动态 Agent 分配**
   - 根据代码类型动态选择审查员（前端代码 → 前端审查员）
   - 根据上一轮发现动态增加审查员（安全问题多 → 加强安全审查）

3. **交叉验证**
   - 审查员 A 发现的问题，让审查员 B 判断是否同意
   - 多数通过 → 确认问题，少数通过 → 标记"待确认"

4. **增量审查**
   - 只审查 git diff 的变更
   - 对比上一版本的审查结果，追踪问题修复状态
