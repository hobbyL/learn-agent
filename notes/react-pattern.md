# ReAct 模式详解

ReAct（Reasoning and Acting）是 Agent 领域最重要的设计模式之一。

---

## 核心思想

**交替进行推理（Reasoning）和行动（Acting）**，而非分离思考和执行。

---

## 为什么需要 ReAct？

### 传统方法的问题

**方法1：只有行动（Action-only）**
```
用户：帮我查一下北京的天气
Agent：[直接调用天气API]
问题：没有推理过程，缺乏可解释性
```

**方法2：只有推理（Reason-only）**
```
用户：帮我查一下北京的天气
Agent：我需要调用天气API，然后返回结果
问题：只说不做，没有实际行动
```

### ReAct 的优势

```
用户：帮我查一下北京的天气
Agent：
  Thought: 用户想知道北京的天气，我需要调用天气API
  Action: call_weather_api(city="北京")
  Observation: {"city": "北京", "weather": "晴", "temp": 25}
  Thought: 已获取天气信息，现在返回给用户
  Action: return_to_user("北京今天晴天，气温25°C")
```

**优势**：
- 推理过程可见
- 行动有依据
- 错误易追踪
- 可中途调整

## ReAct 循环

```
┌──────────────────────────────────┐
│   用户输入（Task）                  │
└──────────┬───────────────────────┘
           │
           ▼
    ┌──────────────┐
    │  Thought     │  ← 推理：分析当前状态
    │  (推理)      │    决定下一步做什么
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │  Action      │  ← 行动：调用工具
    │  (行动)      │    或生成内容
    └──────┬───────┘
           │
           ▼
    ┌──────────────┐
    │ Observation  │  ← 观察：获取行动结果
    │  (观察)      │
    └──────┬───────┘
           │
           │  是否完成任务？
           ├─── 否 ───▶ 回到 Thought
           │
           └─── 是 ───▶ 返回最终答案
```

## 实际案例

### 案例 1：信息查询

**任务**：查找"OpenAI 成立于哪一年"

```
Thought 1: 我需要搜索 OpenAI 的成立时间
Action 1: search("OpenAI 成立时间")
Observation 1: OpenAI 是一家人工智能研究公司，成立于2015年12月...

Thought 2: 找到了答案，OpenAI 成立于 2015 年
Action 2: finish("2015年")
```

### 案例 2：复杂任务

**任务**：找出"北京和上海哪个城市今天更温暖"

```
Thought 1: 需要分别查询北京和上海的天气
Action 1: call_weather_api(city="北京")
Observation 1: {"city": "北京", "temp": 25}

Thought 2: 北京是25°C，现在查上海
Action 2: call_weather_api(city="上海")
Observation 2: {"city": "上海", "temp": 28}

Thought 3: 上海28°C，比北京25°C更温暖
Action 3: finish("上海今天更温暖，28°C（北京25°C）")
```

## Prompt 模板

```
你是一个能使用工具的 AI Agent。

你有以下工具可用：
- search(query): 搜索信息
- calculate(expression): 计算数学表达式
- get_weather(city): 获取天气信息

请使用以下格式回答：

Thought: [你的推理过程]
Action: [工具名称(参数)]
Observation: [工具返回的结果]
... (重复 Thought/Action/Observation 直到任务完成)
Thought: 我现在知道最终答案了
Final Answer: [最终答案]

开始！

Question: {user_question}
Thought:
```

## 实现要点

### 1. 解析 LLM 输出
需要从 LLM 的输出中提取：
- Thought（推理）
- Action（行动）
- Final Answer（最终答案）

### 2. 工具调用
- 解析 Action 中的工具名称和参数
- 执行工具
- 将结果作为 Observation 反馈给 LLM

### 3. 循环控制
- 设置最大步数（防止无限循环）
- 检测是否出现 "Final Answer"
- 处理工具调用失败

---

## 常见问题

### 1. Agent 陷入循环
**原因**：重复相同的 Thought/Action
**解决**：
- 在 prompt 中添加"不要重复之前的行动"
- 检测循环并强制终止

### 2. 工具调用错误
**原因**：LLM 生成的参数格式不对
**解决**：
- 在 prompt 中给出工具使用示例
- 添加参数验证和错误提示

### 3. 推理不充分
**原因**：Thought 太简短，缺乏逻辑
**解决**：
- 在 prompt 中要求"详细说明推理过程"
- 使用 Chain-of-Thought 技巧

---

## 进阶话题

### 1. Self-Consistency
多次运行 ReAct，选择最一致的答案

### 2. ReAct + Reflection
在每个 Action 后增加反思步骤，检查行动是否合理

### 3. Multi-Agent ReAct
多个 Agent 各自执行 ReAct，然后协作

## 从实践中学到的教训

> 以下经验来自 03-react-agent 项目的实际实现与调试，是对上方理论的实战校验。

### 1. ReAct 的核心价值是"幻觉可审计"，而非"让 LLM 不犯错"

ReAct 不会神奇地让 LLM 变得更准确——弱模型依然会跳步、编数据。
ReAct 真正的贡献是：**把错误从"黑盒里不知道的错"变成"推理链里能定位的错"**。

- **Direct 模式**：只看到一个错误答案"1.37 倍"，无从知道哪步出了问题
- **ReAct 模式**：能精确定位到"Step 3 的 Thought 里月影王国面积=8500 是幻觉，
  因为从未有 Observation 返回过这个数字给月影王国"

错误有迹可查，才能针对性地修复（换模型、改 prompt、加护栏）。
这对调试、优化和建立信任都有巨大价值。

### 2. LLM 跳步 bug：把"计划写进 Thought"当成"已经执行"

**现象**：LLM 在 Thought 里写"接下来查月影王国面积"，然后**没有执行 Action**
就直接给出 Final Answer，月影王国面积是凭空编出来的。

```
Thought: 已知星辰王国面积 8500。接下来需要查月影王国的面积。
Final Answer: 星辰王国面积是月影王国的 1.37 倍（8500 / 6200）
```

Step 3 的 Observation 里从来没出现过 6200——这是经典的"计划即执行"幻觉。

**对策 1（纯 prompt，无效）**：加"绝对禁止编造数据"约束 → 弱模型忽视，照样跳步。

**对策 2（代码护栏，有效）**：
检测 Final Answer 前的 Thought 是否含有计划性语言（"接下来"、"还需要"、"然后查"等），
若有则拒绝该 Final Answer，并附上已收集 Observation 摘要，让 LLM 补完计划步骤：

```python
# 拒绝消息模板
f"你的 Thought 里提到了「{hint}」等计划，但你还没有执行这些步骤就给出了 Final Answer。\n\n"
f"你目前已收集的数据：\n{collected_text}\n\n"
f"请继续执行计划中尚未完成的工具调用，不要重新查询你已经有的数据。"
```

**关键**：附上已收集数据摘要是必要的——否则 LLM 看到"请继续"后会"失忆"，
从头重新查已有数据，浪费步数直至 max_steps 耗尽。

### 3. 模型能力是 ReAct 效果的天花板（ReAct 是放大镜）

同样的框架，不同模型表现天差地别：

| 场景 | 强模型（GPT-4o 级） | 弱模型（推理能力不足） |
|------|-------------------|----------------------|
| 跳步 | 老实调工具，不跳步 | 频繁跳步，计划不执行 |
| 实体追踪 | 正确区分多个实体 | Thought 内实体混淆（张冠李戴） |
| 格式遵守 | 稳定输出标准格式 | 随机加 markdown 加粗、漏字段 |

弱模型甚至会在 Thought **内部**产生幻觉——把查到的"星辰王国面积=8500"，
在下一步 Thought 里说成"月影王国=8500"。这类错误代码护栏无法检测到。

**结论**：ReAct 是放大镜，会把模型的能力上限放大，也会把局限暴露得更清楚。
选对模型，是 ReAct 稳定工作的前提条件。

### 4. ReAct vs Direct 真实性能对比

用同一问题（星辰王国面积是月影王国的多少倍）跑两种模式：

|  | ReAct 模式 | Direct 模式（Function Calling） |
|--|-----------|-------------------------------|
| API 轮次 | 7 步（串行） | 4 步（含并行工具调用） |
| 推理可见 | ✅ 完整 Thought 链 | ❌ 黑盒 |
| 并行调用 | ❌ 一次一个 Action | ✅ 一次 API 可多个 tool_calls |
| 适用场景 | 调试、可解释性、学习 | 生产效率、低延迟 |

ReAct 的串行本质让它在多步骤问题上步数更多；Direct 的并行能力让它更高效。
两者各有取舍，不是谁替代谁的关系。

### 5. 虚构数据是验证工具依赖性的标准设计模式

用真实数据（真实国家、真实历史）测试 ReAct 时，LLM 会凭训练记忆直接回答，
跳过工具调用——无法区分"LLM 调工具推理出来的"和"LLM 背出来的"，
测试失去意义。

虚构数据（如"星云大陆"的月影王国面积=5200）彻底消除干扰：
LLM 训练数据里根本没有这个数字，**必须**调工具才能作答。

**结论**：设计 ReAct 学习/测试项目时，优先使用领域专属的虚构知识库，
消除 LLM 先验知识的干扰，才能真正验证 Agent 的工具调用行为。

### 6. 容错设计是 ReAct 循环的必要组成

格式解析失败（LLM 没有输出标准的 `Thought:/Action:/Final Answer:` 结构）
不能让程序崩溃，要给 LLM 一次改正机会。

**重试提示模板**（告知正确格式，并提醒不要用 markdown 加粗）：

```
格式错误：你的输出不符合要求的格式。请严格按照以下格式输出：

Thought: <你的思考>
Action: <工具名>
Action Input: {"参数": "值"}

或者如果你已经有足够信息：

Thought: <总结>
Final Answer: <最终答案>
```

**"最多 2 次重试"策略**：
- 第 1 次失败 → 发送上述格式提示，让 LLM 重试
- 第 2 次失败 → 放弃该步骤，记录错误后终止循环

这个策略平衡了两个目标：
- ✅ 给 LLM 改正机会（偶发格式漂移是正常的）
- ✅ 避免无限重试消耗 token（连续失败说明模型无法稳定遵守格式）

### 7. Function Calling 是工程化的 ReAct

OpenAI 的 `tool_calls` 帮你做了格式解析（省去写正则）、保证了工具调用的原子性，
但代价是**推理过程不可见**——Thought 被封装在模型内部，只看到 Action 和结果。

手动实现 ReAct 文本解析，保留了完整的可观测性，代价是要处理：
- LLM 输出格式不一致（markdown 加粗、字段缺失）
- 需要自己写容错重试逻辑
- 无法利用并行 tool_calls 提升效率

**取舍**：
- 工程生产项目 → Function Calling（效率 + 可靠性）
- 学习 / 可解释性 / 调试场景 → ReAct 文本格式（透明度 + 可审计）
