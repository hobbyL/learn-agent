### 学到的内容

- **OpenAI Function Calling 工作原理**：
  1. 把工具 JSON Schema 传给 LLM
  2. LLM 返回 tool_calls 请求（只是请求，不执行）
  3. 我们的代码执行工具，把结果以 role="tool" 消息返回
  4. LLM 根据结果继续推理，直到输出最终答案
- **messages 列表是 Agent 的记忆**：每次 API 调用都把完整历史传进去，LLM 本身是无状态的
- **AST 白名单安全计算**：比 eval() 更安全，防止代码注入

### 核心循环压力测试（验证完成后的额外探索）
为确认是否真正理解 Agent 核心循环（而非只是工具调用），针对 01 做了三类压力测试：

1. **多轮链式调用**：输入"今天杭州天气，再把温度从摄氏转华氏"
   - 第1轮 get_weather → 29.18°C；第2轮 unit_converter（把 29.18 喂进去）→ 84.5°F；第3轮 LLM 综合两步给最终回答
   - 收获：**工具链是 LLM 自主串起来的**，代码里没有任何"天气之后转温度"的逻辑；`messages` 数量 2→4→6 增长就是记忆累积，第2轮能拿到 29.18 正是因为第1轮结果留在了 messages 里
2. **错误中断与优雅降级**：输入"查不存在的城市'阿斯加德'天气再转华氏"
   - 第1轮 get_weather → error；第2轮 LLM 读懂 error 后**主动终止任务链**，没去做不可能的第二步
   - 收获：**工具的 error 信息是给 LLM 做决策用的**，不只是给人看。明确的错误（"未找到城市…拼写有误"）让 LLM 判断重试无意义；若只返回 `{"error":"request failed"}` 则可能误判为可重试而绕圈撞 max_iterations
3. **撞迭代上限**：仅理解原理，未刻意复现（需人为制造"看似可重试、实则永久失败"的场景）

### 学到的内容
- **prompt 是 Agent 行为最强的杠杆**：发现"问 2¹⁰ 却乱调 hash 工具、还无视结果自己心算"的缺陷后，**一行逻辑代码没改**，仅在 SYSTEM_PROMPT 加"能力边界 + 工具结果必须使用"两条约束，行为立刻扭转——成为"改 prompt 看 Agent 行为变化"实验的正向验证
- **能力边界判断比想象中细腻**：LLM 不仅能判断"有没有工具"，连"有相关工具但能力不够"（color_converter 在、但不支持 YUV）都能正确识别为"调了也没用"，转而诚实说明
- **但这是概率护栏，不是确定性兜底**：今天表现完美，换问法/换模型仍可能翻车，真正的确定性约束要留到 03/04
- **手搓 Schema 的边界 = Pydantic 的价值**：手搓版只能"告诉 LLM"枚举范围，无法在运行时强制校验非法值——这正是下半场引入 Pydantic 要补的洞

### 遇到的问题
- **幻觉式工具调用**：问"计算 2 的 10 次方"（02 无计算器工具），LLM 硬调 `hash_generator`，又无视哈希结果自行心算出 1024（蒙对但过程错）→ 根因：SYSTEM_PROMPT 只说"必须用工具"却没说"没有匹配工具时该如实拒绝" → 修复：新增「能力边界」「工具结果必须被使用」两条最高优先级约束，并定调"诚实+尽力"语气

### 学到的内容
- **Pydantic 的核心价值不是"少写代码"，是"运行时强制校验"**：手搓版能生成 Schema 告诉 LLM，但拦不住 LLM 传错值；Pydantic 在函数调用前就拦截
- **model_json_schema() 输出需要适配**：Pydantic 会加 title、可能有 $defs，直接喂 OpenAI 有冗余，需要一层 `_clean_schema()` 去掉
- **ValidationError 结构设计精良**：field 路径 + msg + input，天然适合转成 LLM 能理解的纠错提示
- **类型宽容转换是隐藏优势**：LLM 传 `"16"` 给 int 字段，Pydantic 自动转换而不是报错，减少无谓失败
- **配置切换 + 并存对照是好的学习模式**：不是"替换"而是"并存"，随时切换体验差异
