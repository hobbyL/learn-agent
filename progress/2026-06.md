### 核心循环压力测试（验证完成后的额外探索）
为确认是否真正理解 Agent 核心循环（而非只是工具调用），针对 01 做了三类压力测试：

1. **多轮链式调用**：输入"今天杭州天气，再把温度从摄氏转华氏"
   - 第1轮 get_weather → 29.18°C；第2轮 unit_converter（把 29.18 喂进去）→ 84.5°F；第3轮 LLM 综合两步给最终回答
   - 收获：**工具链是 LLM 自主串起来的**，代码里没有任何"天气之后转温度"的逻辑；`messages` 数量 2→4→6 增长就是记忆累积，第2轮能拿到 29.18 正是因为第1轮结果留在了 messages 里
2. **错误中断与优雅降级**：输入"查不存在的城市'阿斯加德'天气再转华氏"
   - 第1轮 get_weather → error；第2轮 LLM 读懂 error 后**主动终止任务链**，没去做不可能的第二步
   - 收获：**工具的 error 信息是给 LLM 做决策用的**，不只是给人看。明确的错误（"未找到城市…拼写有误"）让 LLM 判断重试无意义；若只返回 `{"error":"request failed"}` 则可能误判为可重试而绕圈撞 max_iterations
3. **撞迭代上限**：仅理解原理，未刻意复现（需人为制造"看似可重试、实则永久失败"的场景）

### 学到的内容
- **ReAct 的核心价值是"幻觉可审计"**：Direct 模式只看到错误答案；ReAct 模式能从 Thought 链里找到哪一步出了幻觉——错误有迹可循，可调试
- **LLM 跳步 bug 是 ReAct 经典陷阱**：LLM 在 Thought 里写"接下来要查 X"，然后直接给 Final Answer 编造数字——"计划写进 Thought"不等于"Action 已执行"
  - 对策 1：强化 System Prompt（对弱模型效果有限）
  - 对策 2：代码层检测计划性关键词，拒绝该 Final Answer，同时附上已收集数据摘要避免 LM 失忆
- **模型能力是框架的天花板**：同一套 ReAct 框架，强模型（gpt-4o 级）7 步完美完成；弱模型甚至会在 Thought 内部产生幻觉（拿错 Observation 数据），代码层难以拦截
- **ReAct vs Direct 效率差异**：Direct 可以一次 API 调用并行执行多个工具（4 步）；ReAct 本质上串行（7 步）；但 ReAct 的推理透明度是 Direct 无法比拟的
- **虚构知识库是测试 Agent 的利器**：LLM 训练数据没有"星云大陆"，强迫它真正走 Thought→Action→Observation 而非凭记忆作答
- **AST 白名单安全计算**：calculate 工具用白名单 AST 节点替代 `eval()`，防止任意代码执行

### 遇到的问题
- **search() 去重 bug**：内层 `break` 只跳出内层 `for item in value`，外层继续迭代导致同一实体多次添加 → 修复：`matched = False` 标记，找到后设 `True` 并 `break` 外层
- **LLM 输出 markdown 加粗导致正则失效**：LLM 输出 `**Thought:**` 而非 `Thought:` → 修复：正则加 `\*{0,2}` 前后缀兼容
- **跳步检测后 LLM 失忆**：被拒绝后 LLM 忘记已有数据，重新查询导致步数耗尽 → 修复：拒绝消息中附上已成功 Observation 摘要
- **MIN_OBSERVATIONS 守卫过于粗暴**：要求最少 2 次观察才能给答案，导致单步问题被强制多调工具 → 删除，改用更精确的计划关键词检测

### 学到的内容
- **Reflexion 的本质**：不是"重试"，而是"带记忆的反思式重试"——反思摘要注入 system prompt 尾部，下一轮 LLM 第一步就能看到"上次哪里犯了错"
- **内层 Actor 要故意简化**：护栏越强，Agent 越不会犯错，反思机制就越没用武之地。04 特意去掉 03 的跳步检测，让错误自然发生
- **评估器是 Reflexion 的瓶颈**：Ground Truth 需预定义答案（学习/测试用）；LLM Judge 灵活但有主观性（可能误判）。真实场景最好用客观信号（测试通过/失败、schema 校验、代码执行结果）
- **双轨评估器的参数签名不同**：GroundTruth.evaluate(question, answer, ground_truth) vs LLMJudge.evaluate(question, answer, steps)，组合调用时需要 isinstance 路由
- **MAX_TRIALS=3 的实验依据**：Reflexion 论文表明 2-3 轮后改进饱和；超过 3 轮通常说明问题超出模型能力或评估器有问题
- **结构性约束比策略更根本**：部分任务在 MAX_STEPS 不足时，无论反思多少次都无法完成——反思改变策略，但改变不了步数上限。配置 MAX_STEPS=15 解决 q07 路径需要 ≥12 步的问题

  - 02: 工具注册+Schema+Pydantic 校验
  - 03: ReAct 推理链可审计性（硬护栏容错）
  - 04: Reflexion 试错改进循环（软反思容错）

### 学到的内容
- **流式 ReAct 的核心挑战是标签跨 chunk**：`\nTho` + `ught:` 两个 chunk 才能确认是 `Thought:` 标签，需要 buffer 暂存等待确认
- **ActionReady 只在 flush() 时发出**：ReAct 每步输出 `Action + Action Input` 后流就结束，不会跟新 section，必须在 flush() 里检查状态
- **状态机 vs 正则**：非流式用正则一次解析简单直接；流式用状态机逐字解析，复杂度更高但实现了实时展示
- **steps 一致性验证**：compare 模式证明两种实现逻辑等价（同一问题步数相同，答案一致）
- **工具错误提示设计很重要**：第3步查"驻扎站"失败，工具返回"可用属性"列表帮助 Agent 自动纠正为"驻站"

### 遇到的问题
- **chunk.choices 空列表**：某些 streaming chunk 的 choices=[]，直接索引 [0] 报错 → 加 `if not chunk.choices: continue` 判断
- **ActionReady 不触发**：最初设计在切换 section 时发出，但 ReAct 每步流结束就停，没有下一 section → 改为 flush() 里检查
- **属性名不匹配**：Agent 用"驻扎站"查询失败，工具返回"可用属性: 驻站"后自动纠正
