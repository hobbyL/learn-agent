## 🧭 待评估事项（滚动规划，不急于现在动手）

完成 01 后发现的计划改进点，先存档，按"做完一个阶段再复盘"的节奏逐步处理：

### 项目边界待澄清
- **01 / 02 边界重叠**：01 (simple-agent) 已完整实现工具调用，02 (tool-calling) 需重新定位为"工具系统架构"（注册机制 + 参数模型），而非重做工具调用。
- **03 / 04 可能合并**：ReAct (03) 与 Reflection (04) 本就是递进关系（Reflexion = ReAct + 反思），警惕重复，考虑合并或明确各自侧重。
- **错误处理是否单独立项**：07 (error-recovery) 的内容其实贯穿每个项目（参见 notes 中"error 信息影响循环"），可能更适合作为横切主题而非独立项目。

### 候选补充主题（现代 Agent 绕不开，但原计划缺失）
- **上下文管理 / context 压缩**：长任务下 context 必然撑爆，如何摘要/裁剪历史是生产级 Agent 第一痛点（memory-system 沾边但不等同）。
- **结构化输出 & 可靠解析**：让 LLM 稳定吐出可被代码消费的格式，比工具调用更底层的可靠性基础。
- **流式输出 (streaming)**：体验与可观测性关键，纯实现层但绕不开。
- **MCP（Model Context Protocol）动态工具发现**：对照手搓 registry（编译时静态绑定），MCP 是运行时绑定——Agent 启动后向 Server 询问"你有什么工具"。建议时机：阶段 2 过半后（做完 memory/task-planning），届时对工具系统理解够深，MCP 的协议层价值感最强。
- **框架对照节点**：方法论是"先手写再学框架"，但 12 个项目全是手写，缺一个明确"对照 LangGraph / Pydantic AI / OpenAI Agents SDK"的节点，用来验证自己手写的理解。

> 原则：方向已对（原理→系统→多 agent），下一步已清晰（03）。不过早精修后续项目的细节，
> 对后续项目的理解会在做完 03、04 后大幅提升，届时滚动调整。

## 💡 学习方法

1. **项目驱动**：每个阶段都有可交付的项目
2. **原理优先**：先手写理解原理，再学习框架
3. **记录踩坑**：在项目 `notes.md` 中记录问题和解决方案
4. **定期提炼**：将项目笔记中的通用知识提炼到主目录 `notes/`
5. **阶段复盘**：完成一个阶段后写总结，沉淀经验
