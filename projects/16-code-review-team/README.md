# 16-code-review-team：代码审查团队（分工协作）

多 Agent 分工协作模式，4 个审查员并行审查代码 + 主审汇总。

## 特性

- **两层协作架构**：4 个专业审查员（安全/性能/架构/规范）并行执行 → 主审汇总去重
- **结构化输出**：审查员使用 json_schema 强制输出（复用项目 09 strict 模式）
- **虚构代码库**：4 段有预设问题的代码，覆盖不同审查维度
- **召回率验证**：对比预设问题和实际发现，验证审查员准确性
- **ANSI 着色**：5 个 Agent 各有独立颜色

## 运行

```bash
# Demo 模式（审查代码片段 A：用户认证模块）
python main.py --demo

# Interactive 模式（选择代码片段或输入自定义代码）
python main.py --interactive
```

## 审查团队

| Agent | 角色 | 关注点 | 颜色 |
|-------|------|--------|------|
| 🛡️ 安全审查员 | Security Reviewer | SQL 注入、硬编码密钥、权限校验缺失 | 红色 |
| ⚡ 性能审查员 | Performance Reviewer | O(n²) 循环、重复计算、阻塞操作 | 黄色 |
| 📐 架构审查员 | Architecture Reviewer | 职责不清、耦合过紧、错误处理缺失 | 蓝色 |
| 📝 规范审查员 | Code Style Reviewer | 命名不规范、注释缺失、代码重复 | 青色 |
| 👨‍⚖️ 主审 | Lead Reviewer | 汇总去重、排优先级、输出最终报告 | 绿色 |

|------|------|---------|------|

## 协作流程

```
代码片段
    ↓
┌─────────────────────────────────────────┐
│  第一层：4 个审查员并行执行              │
│  ┌──────────────┐  ┌──────────────┐   │
│  │ 🛡️ 安全审查员│  │ ⚡ 性能审查员│   │
│  └──────────────┘  └──────────────┘   │
│  ┌──────────────┐  ┌──────────────┐   │
│  │ 📐 架构审查员│  │ 📝 规范审查员│   │
│  └──────────────┘  └──────────────┘   │
│         ↓ json_schema 强制输出           │
└─────────────────────────────────────────┘
    ↓
┌─────────────────────────────────────────┐
│  第二层：主审汇总                        │
│  👨‍⚖️ 主审：去重 + 排优先级 + 最终报告 │
│         ↓ 自然语言总结                   │
└─────────────────────────────────────────┘
    ↓
最终审查报告
```

## 技术亮点

1. **json_schema strict 模式**：审查员输出 100% 结构化，复用项目 09 经验
2. **两层协作模式**：并行执行 → 汇总决策，和项目 15 的"辩论投票"形成对比
3. **召回率验证**：预设问题 vs 实际发现，量化审查员准确性
4. **去重逻辑**：主审能识别多个审查员报告的相同问题
5. **优先级排序**：P0（致命）> P1（严重）> P2（建议）

## 与项目 15 的对比

| 维度 | 15-辩论投票 | 16-分工协作 |
|------|-----------|-----------|
| 目标 | 达成共识（选一个） | 全面覆盖（找出所有问题） |
| Agent 关系 | 对立（各持己见） | 互补（各管一面） |
| 信息流 | 交叉（看到彼此发言） | 独立（各审各的） |
| 输出 | 投票结果 | 聚合报告 |
| 阶段 | 三阶段辩论 | 两层协作 |
