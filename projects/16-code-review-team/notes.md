# 项目 16 笔记：代码审查团队（分工协作）

## 实现日期

2026-07-20

## 核心收获

### 1. 多 Agent 协作模式二：任务分解 + 并行聚合

和项目 15 的"辩论投票"完全不同的协作模式：

**项目 15（辩论投票）**：
- 同一问题，不同立场
- Agent 之间**对立**（互相质疑）
- 信息**交叉**（看到彼此发言）
- 目标：达成**共识**

**项目 16（分工协作）**：
- 同一任务，不同维度
- Agent 之间**互补**（各管一面）
- 信息**独立**（各审各的）
- 目标：全面**覆盖**

### 2. 两层协作架构

```
第一层：并行执行层（4 个专业审查员）
    ↓ 各自独立审查
第二层：汇总决策层（1 个主审）
    ↓ 去重 + 排序 + 输出
```

关键设计：
- 第一层用 json_schema 强制输出（结构化，方便聚合）
- 第二层用自然语言输出（可读性，面向人类）
- 主审的价值：去重（多人报同一问题）+ 排优先级（P0/P1/P2）

### 3. 结构化输出的工程价值

**审查员为什么必须用 json_schema？**
- 主审需要精确解析：文件名、行号、严重级别、分类
- 如果审查员输出自然语言，主审很难去重（同一问题不同表述）
- 结构化 = 可编程聚合

**主审为什么可以用自然语言？**
- 最终报告面向人类开发者，不需要再次解析
- 自然语言总结更易读

### 4. 虚构代码库的设计技巧

每段代码的预设问题需要：
- **单一维度集中**：代码 A 主要是安全问题，代码 B 主要是性能问题
- **问题清晰可验证**：有明确的文件名、行号、问题类型
- **覆盖典型场景**：SQL 注入、O(n²)、职责不清、命名混乱等常见问题

这样设计的好处：
- 方便验证审查员是否找到（召回率）
- 避免维度重叠（多个审查员报同一问题是去重测试，不是设计目标）

### 5. 去重逻辑的本质

主审怎么去重？看两个维度：
- **文件名 + 行号**：同一位置的问题大概率是同一个
- **问题描述语义**：即使行号不同，描述相似的问题也可能重复

项目中主审用 LLM 做语义去重，比规则匹配更灵活。

## 遇到的问题

### 1. 初次实现未遇到大问题

项目 15 和项目 09 的经验直接复用：
- json_schema strict 模式（项目 09）
- 并行执行 + 回调展示（项目 15）
- ANSI 着色（项目 15）

### 2. API 调用成本

5 个 Agent（4 审查员 + 1 主审）= 5 次 API 调用，每次审查一段代码需要几分钟。

优化方向（未实现）：
- 用 asyncio 真正并行（4 个审查员同时调用）
- 减少审查员数量（3 个也能展示协作）

## 技术细节

### 1. json_schema 强制输出

```python
schema = get_json_schema(ReviewResult)

response = client.chat.completions.create(
    model=model,
    messages=messages,
    response_format={
        "type": "json_schema",
        "json_schema": {
            "name": "review_result",
            "strict": True,
            "schema": schema
        }
    }
)
```

### 2. 主审的输入格式

把 4 个审查员的结构化结果转成文本，注入主审的 messages：

```python
reviews_text = "\n\n".join([
    f"=== {result.reviewer} ===\n"
    f"发现 {len(result.findings)} 个问题\n"
    f"问题列表：\n" +
    "\n".join([f"- [{f.severity}] {f.file}:{f.line} - {f.description}" 
               for f in result.findings])
    for result in reviewer_results
])
```

主审看到的是"4 份报告的文本汇总"，不是原始 JSON。

## 可能的扩展

1. **真正的并行执行**：用 asyncio 让 4 个审查员同时调用 API
2. **更多审查维度**：可测试性、国际化、无障碍性等
3. **审查员之间交叉验证**：安全审查员发现的问题，让性能审查员判断是否有性能影响
4. **增量审查**：只审查 git diff 的变更部分
5. **自动修复建议**：不仅指出问题，还生成修复后的代码

## 总结

项目 16 是多 Agent 协作的第二种典型模式（任务分解 + 并行聚合），和项目 15 的"辩论投票"形成完整对比：

- **15 是收敛型**：多个观点 → 一个决策
- **16 是发散型**：一个任务 → 多个维度 → 汇总报告

两种模式都是实际 Agent 系统的常见架构。
