# 15-multi-agent-debate 实现笔记

## 踩坑记录

### 1. .env 变量名不一致

**问题**：.env 里是 `OPENAI_MODEL`，代码里读 `MODEL_NAME`，默认值 `deepseek-v4-0324` 已下线。

**修复**：统一读 `OPENAI_MODEL`，默认值改为 `deepseek-v4-flash`。

**教训**：每个项目的 .env 变量名要和代码一致，最好在 create_client() 里统一处理。

### 2. 投票 JSON 解析容错

**问题**：LLM 输出的投票 JSON 格式不稳定，可能是：
- 标准 ```json 代码块
- 裸 JSON 嵌在文本中
- 完全没有 JSON

**解决**：三级降级策略：
1. 正则匹配 ```json 代码块
2. 正则匹配裸 `{"vote":...}` 
3. 启发式提取（搜索"投票：XX"/"最终选择 XX"）
4. 最后兜底：文中最后出现的星球名

### 3. 推荐提取的启发式

第一轮推荐提取用关键词匹配（"推荐XX"、"选择XX"、"首选XX"），如果都没命中就取最后一次提到的星球名。这个方法有误判风险但实际效果还行。

更好的方案是用 json_schema 强制输出，但本项目侧重辩论编排，投票解析不是核心，启发式够用。

### 4. 信息隔离 vs 信息注入的实现

关键是 `run_agent()` 的 `context` 参数：
- 第一轮：不传 context → Agent 只看 system prompt + 阶段提示 + 知识库
- 第二轮：context = 格式化的其他人第一轮发言
- 第三轮：context = 所有轮次所有人的发言

信息隔离不是靠复杂的权限控制，而是靠**不传入不该看到的信息**。

### 5. 工具调用轮数

`max_turns=10` 是 ReAct 的最大工具调用轮数。通常 Agent 只需 2-4 轮工具调用就能收集够信息。超过 10 轮强制追加 "不要再用工具" 的指令获取最终回答。

## 设计思考

### 多 Agent 协作的核心模式

本项目验证了一个典型的多 Agent 协作模式：

```
独立推理 → 交叉质疑 → 收敛共识
```

这和单 Agent 多轮对话的根本区别：
- 单 Agent：一个视角，自己质疑自己
- 多 Agent：多个视角，真正的交叉检验

### 共享 + 专属工具的价值

比起"所有 Agent 共享所有工具"，混合方案更贴近真实场景：
- 共享工具 = 基础能力（查询数据）
- 专属工具 = 专业能力（领域分析）

这让每个 Agent 有独特的信息来源，辩论更有实质性分歧。

### 立场变化追踪

对比 round1_recommendation vs round3_vote 是多 Agent 辩论的特有产物：
- 全票一致 = 共识强
- 有人改变立场 = 辩论有效，被说服了
- 无人改变 = 各持己见，需要更多轮或仲裁机制

## 对比项目 04（Reflexion）

| 维度 | 04-agent-reflection | 15-multi-agent-debate |
|------|--------------------|-----------------------|
| Agent 数量 | 1（+ 评估器） | 3 个独立角色 |
| 反馈来源 | 自我反思 + 外部评分 | 其他 Agent 的质疑 |
| 信息流 | 纵向（历史积累） | 横向（跨角色交叉） |
| 收敛机制 | 评分达标 | 投票表决 |
| 立场变化 | 不适用 | 可追踪 |
